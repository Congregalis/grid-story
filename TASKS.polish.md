### 设计要点

- **路由形态**:推荐 `/books`(书架)、`/books/:id`(工作台)、`/books/:id/bible`、`/books/:id/outline`、`/books/:id/writing`、`/books/:id/settings`。bookId 从 URL 拿,而非现在的 `useBookId()` localStorage。
- **「下一步」状态机**:写一份 `nextStep(book, stats) → { label, to, urgency }`,在 `App.tsx` 顶部读一次,所有页面共用。状态机分支(草拟):
  - charter 完成度 == 0 → "先填创作设定"
  - 角色 == 0 → "建第一个角色"
  - outline 节点 < 4 → "搭大纲"
  - 无 chapter → "新建第一章"
  - 有 draft → "继续写《章名》"
  - 有 review → "审定章节"
  - 全 final → "新建下一章"
- **不要**重做 PixelKit 组件,二级导航用现有 NavLink 风格扩展即可
- 工作台主区先放空骨架占位,P5 / P3 进来再填

**背景**:TipTap 默认就是个网页输入框。写小说重度依赖键盘、自动保存、专注感。改了不主动点保存就会丢、不能 F11 沉浸、没快捷键。
**范围内**:专注模式 / 自动保存 + 状态指示 / 全局快捷键。
**不做**:富文本扩展(标题/列表已够用)、协作。

| ID         | 任务                                                              | 估时  | 验收                                                  |
| ---------- | ----------------------------------------------------------------- | ----- | ----------------------------------------------------- |
| T2.6.P4.1  | 专注模式(`F11` 或界面按钮触发,`Esc` 退出)                       | 0.2d  | 进入后:NavBar / 左右栏 / 顶部条全部隐藏,只剩文本 + 字数;再按一次还原 |
| T2.6.P4.2  | 自动保存:onUpdate debounce 5s 写新版本                           | 0.3d  | 走现有 chapter 多版本 API,版本号合理(不要每 5s 都开新版,合并到同一 draft 版本下) |
| T2.6.P4.3  | 顶部状态指示灯:已保存 / 保存中 / 离线                            | 0.15d | 三态切换;离线时(navigator.onLine == false 或保存连续失败)红点 + 提示"已暂存到本地" |
| T2.6.P4.4  | 全局快捷键:`Cmd+S` 强制存版本 / `Cmd+J Cmd+K` 上下章 / `Cmd+/` 切大纲 / `?` 帮助层 | 0.25d | 帮助层列出所有快捷键;mac/win 自动适配 Cmd/Ctrl    |
| T2.6.P4.5  | 验收                                                              | 0.1d  | 关页面 → 重开 → 草稿仍在;断网模拟 → 状态灯红,本地不丢失 |

### 实现提示

- "本地暂存"用 IndexedDB 或 localStorage(章节字数大,localStorage 可能爆),建议 `idb-keyval` 一行
- 自动保存的"合并到同一 draft 版本"指:同一 session 内多次 onUpdate 不创建新版本,只更新 draft;切章节或显式 Cmd+S 时落新版本
- 快捷键库用原生 `keydown` 监听即可,不引 mousetrap

## T2.6.P6 · 文中设定高亮(0.5d)

**背景**:右栏实体卡显示了当前章节涉及的设定，但读者看不到"文中的哪个词对应哪个设定"。在正文里把实体名按类型染色，一眼能看出角色/地点/组织/物品的分布。

**不做**:NER 实体链接（留 T3）、跨段落关系标记。

## T2.4.P2 · AI 选区改写(1d)

**背景**:目前 AI 只能「整章首稿」，用户接受后想局部调整（"这段描写太干了"、"改成更紧张的氛围"）没有入口。需要支持选中文本 + 指令改写，以及无选区时的全文修订。

**范围内**:选区改写 UI（浮动工具栏或快捷键）、后端 `/agent/writing/rewrite` endpoint、Prompt 模板、改写结果走候选面板。
**不做**:多轮对话式迭代（留 T3.4 RewriteAgent）、token-level diff（留 T3.8）。

| ID         | 任务                                                              | 估时  | 验收                                                  |
| ---------- | ----------------------------------------------------------------- | ----- | ----------------------------------------------------- |
| T2.4.P2.1  | 选区改写 UI:选中文本后出现浮动按钮「AI 改写」;弹小输入框写指令    | 0.25d | 有选区时浮动按钮出现;无选区时仅显示全文修订入口        |
| T2.4.P2.2  | 后端 endpoint:`POST /agent/writing/rewrite`                        | 0.2d  | 接收 `{bookId, selectedText, instruction, contextText}` 返回 `{rewritten}` |
| T2.4.P2.3  | Prompt 模板:`packages/prompts/writing/rewrite.v1.md`                | 0.15d | 包含当前章节上下文、Charter 设定摘要、用户指令          |

- 选区浮动按钮用 TipTap 的 `FloatingMenu` 或自写绝对定位 `<div>`（监听 `editor.state.selection` 变化）
- 无选区时可以是编辑器上方的一个固定按钮「AI 修订全文」，复用 rewrite endpoint，`selectedText` 传全章内容
- `selectedText` 传给 agent 时应附带前后各一段的上下文（`contextText`），让改写衔接更自然
- Prompt 模板中的 Charter 设定摘要可用现有的 ContextComposer 基础设施

## T2.4.P3 · AI 审稿(ReviewAgent,1d)

**背景**:当前「送审」只是改状态字段，没人真正审。作者需要一份结构化的审稿意见——角色是否一致、情节节奏、文笔问题、改进方向——类似编辑给作者的反馈信。

**范围内**:送审时触发 AI 审稿、结构化审稿结果（一致性/节奏/文笔/建议四维度）、审稿 UI（侧栏卡片列表 + 采纳建议按钮）。
**不做**:自动修改（人审为主，采纳建议交给 P2 选区改写执行）、多轮审稿、评分数字化。

| ID         | 任务                                                              | 估时  | 验收                                                  |
| ---------- | ----------------------------------------------------------------- | ----- | ----------------------------------------------------- |
| T2.4.P3.1  | 后端 endpoint:`POST /agent/writing/review`                         | 0.2d  | 接收 `{bookId, chapterRootId}`，返回 `{issues: [{dimension, severity, quote, comment, suggestion}]}` |
| T2.4.P3.2  | Prompt 模板:`packages/prompts/writing/review.v1.md`                 | 0.2d  | 输入:章节全文+Bible设定+大纲节点+Charter;输出:结构化审稿 JSON |
| T2.4.P3.3  | 审稿 UI:WritingDesk 侧栏/浮层展示审稿结果列表                     | 0.35d | 按维度分组(一致性/节奏/文笔/建议);每条显示原文引用+问题描述+修改建议;「采纳建议」按钮(将建议填入选区改写入口) |
| T2.4.P3.4  | 送审流程改造:送审按钮 → 触发 review + 状态切 review               | 0.1d  | 点击送审后自动调 review endpoint,结果展示在侧栏       |
| T2.4.P3.5  | 验收                                                              | 0.15d | 端到端:写一章→送审→审稿结果出现→点采纳建议→选区改写UI打开并预填指令→改写→接受 |

### 实现提示

- 审稿结果 JSON schema 在 `packages/schema` 中新定义（与其他 schema 共享）
- 审稿 UI 可以用现有右栏或新建一个可切换的 ReviewPanel，避免和实体卡同时占用空间
- 「采纳建议」流程：点击建议 → 在原文中定位相关段落（简单字符串匹配）→ 自动选中 → 打开选区改写 UI 并预填 instruction
- 审稿是异步操作（可能 30s+），UI 需要 loading 态 + 流式输出可选（本期先同步等待）
- 与 T2.4.P2 的关系：审稿建议的"采纳"自然导向选区改写，两者配合使用效果最好
